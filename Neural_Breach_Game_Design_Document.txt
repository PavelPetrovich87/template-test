# Neural Breach: Game Design Document

**Project Name:** Neural Breach  
**Tagline:** *Train your working memory by hacking corporate firewalls.*  
**Platform:** Web (React/Vue), Mobile-ready (responsive CSS)  
**Target Audience:** Cognitive training enthusiasts, productivity-focused users, competitive learners  
**Genre:** Educational Cognitive Training + Idle RPG  

---

## 1. Executive Summary

Neural Breach wraps the scientifically-proven **Dual N-Back** working memory training task inside a cyberpunk hacking narrative. Users are "hackers" attempting to breach firewalls by maintaining focus on a dual-stream memory task while managing a "Connection Stability" resource. The core training mechanic is preserved (two concurrent stimulus streams, adaptive difficulty, immediate feedback), but wrapped in a **Terminal UI** with simple progression systems (Data Shards, Hardware Upgrades, Themes) that reward consistent play without adding cognitive load during training.

**Key Design Philosophy:** The game should feel rewarding, not punitive. Every correct answer is a tiny victory. Every mistake costs you something (Stability, Data Shards), but doesn't break the training flow or data collection.

## 2. Core Gameplay Loop

### 2.1 The Training Session ("Breach Attempt")

**Duration:** 3â€“5 minutes per session (customizable)  
**Structure:** One session consists of multiple *blocks*, each block = 20 trials (1 trial = 1 stimulus pair)

#### The N-Back Task (Unchanged)
- **Visual Stream:** A 3Ã—3 grid. One cell lights up every 2â€“3 seconds.
- **Audio Stream:** A letter (Aâ€“Z) plays every 2â€“3 seconds.
- **Player Input:** Two buttons (or keyboard keys).
  - Button 1: "Match Audio" (press if the current letter matches the letter from *n* steps back).
  - Button 2: "Match Position" (press if the current grid cell matches the cell from *n* steps back).
- **Difficulty:** Starts at n=2. Adapts:
  - **Accuracy â‰¥ 80%:** Increase to n+1.
  - **Accuracy â‰¤ 60%:** Decrease to nâˆ’1.
  - Adapts once per block.

#### New Layer: Connection Stability

**What it is:** A health bar representing your "hacker's connection to the target."

**Mechanics:**
- **Starting Value:** 100% per block.
- **Correct Match:** Stability holds (no change). Player earns +1 Data Shard.
- **Correct Non-Match (Correctly NOT pressing):** Stability holds. Player earns +1 Data Shard.
- **False Alarm (Pressing when no match):** Stability drops by 15%. Player earns 0 Shards. *Visual Feedback:* Screen border flashes red (0.3s CSS animation).
- **Miss (Not pressing when there IS a match):** Stability drops by 10%. Player earns 0 Shards. *Visual Feedback:* Screen border flashes yellow (0.3s CSS animation).
- **Zero Stability Reached:** Block continues (data is still collected for research), but UI enters "Safe Mode":
  - Screen desaturates (opacity: 0.6, grayscale: 50%).
  - No more Data Shards earned for the rest of the block.
  - Stability bar shows a warning badge: "âš  Connection Lost."

**Why:** Stability adds emotional stakes without changing the training. Users feel they are "protecting a resource" rather than just answering questions.

## 3. Resource Economy

### 3.1 Data Shards (Primary Currency)

**What it represents:** Stolen data = reward for successfully maintaining focus.

**Earning Rules:**
- **Base Rate:** 1 Shard per correct trial.
- **Streak Bonus:** 5 consecutive correct trials â†’ Next correct match = 2 Shards.
- **Session Bonus:** Complete all blocks without losing Stability to zero â†’ +5 bonus Shards.
- **Daily Bonus:** If you train 3+ days in a row â†’ Session rewards Ã— 1.2 multiplier.

**Spending:**
- Buy Hardware upgrades (see Section 4).
- Buy cosmetic themes.
- *Future expansion:* Trade for leaderboard entries or seasonal badges.

**Display:** Top-right corner of the main UI. Icon: `ðŸ’¾` next to a number. Persistent across sessions.

## 7. Data & Analytics

### 7.1 What We Track (For Research)

The app logs every trial with:
- Timestamp
- Stimulus (grid position + audio letter)
- N-level at trial
- User response (match / non-match)
- Accuracy (correct / incorrect / false alarm / miss)
- Reaction time (ms)
- Stability at trial start

**Reporting:** Users can view:
- **Session History:** A table of all past sessions (date, duration, peak n-level, avg accuracy).
- **Progress Graph:** Working memory growth over time (x-axis: days, y-axis: peak n-level reached).
- **Personal Best:** "Highest n-level reached: 5" or "Best accuracy: 94%."

### 7.2 Leaderboard (Optional, Social)

**Leaderboard Types:**
- **Weekly Peak N-Level:** Who reached the highest n-level this week?
- **Accuracy Champions:** Who maintained >90% accuracy?
- **Consistency Kings:** Who trained the most days in a row?

**Anonymous by default.** Users can opt-in to show their username.

### 8.2 Key Subsystems

| Subsystem | Responsibility |
| :--- | :--- |
| **Stimulus Generator** | Create random grid + audio sequences respecting n-back rules. |
| **Difficulty Adapter** | Adjust n-level based on accuracy after each block. |
| **Input Handler** | Capture button presses, validate timing, store reaction time. |
| **Feedback Engine** | Update Stability, increment Shard count, queue animations (border flash). |
| **Persistence Layer** | Save session data to backend (or localStorage for MVP). |
| **Shop Engine** | Validate purchases, apply upgrade effects to user state. |
| **Analytics Dashboard** | Compute stats (accuracy %, peak n, trends) for display. |

### 8.3 Frontend Stack Suggestion

Given your React/Vue expertise:
- **Framework:** React (Hooks for state) or Vue 3 (Composition API).
- **Styling:** CSS-in-JS (Styled Components) or Tailwind for rapid theming.
- **State Management:** Zustand or Pinia (lightweight, fast).
- **Storage:** IndexedDB for offline-first session caching; sync to backend on network.
- **Animations:** CSS keyframes only (no GPU-heavy JS animations during trials).

### 8.4 Backend (Optional for MVP)

**Minimal Backend:**
- User authentication (Firebase Auth or similar).
- Session data storage (Firestore or PostgreSQL).
- Leaderboard aggregation.

**For MVP:** Could be entirely client-side (localStorage + export-to-JSON for backup).

| Risk | Mitigation |
| :--- | :--- |
| Users quit due to difficulty spikes. | Stability bar acts as a "safety net." Failsafe Protocol upgrade available. Tutorial session explains difficulty adaptation. |

## Design Notes for Development

1. **Preserve the Science:** Every change to UI/UX must be validated against the core n-back principles. When in doubt, ask: "Does this distract the user from the dual streams?"

2. **Minimize Animation:** CSS-only animations on non-critical elements (borders, text color). No GPU-heavy transforms during trials.

3. **Mobile-First CSS:** The grid and buttons must be touchable and not cramped on a 5-inch phone.

4. **Accessibility:** High contrast (WCAG AA). All interactive elements keyboard-accessible. Screen reader friendly.

5. **Performance:** Session logic should run at 60 FPS. Stimulus timing must be precise (Â±50ms tolerance). Use `requestAnimationFrame` for smooth renders.

6. **Data Export:** Users should be able to export their session history as CSV for personal records or research collaboration.
